# Auto-Update — Design

**Status:** Shipped in v0.1.2 (commit `8356a29`, 2026-04-17).
**Scope:** Cryptographically-signed, one-click updates checked against the GitHub Releases latest-tag endpoint on app startup.

## Goal

On every launch, silently check GitHub for a newer version. If one exists, prompt the user with a native dialog. On accept, download the signed bundle, verify against an embedded public key, install, and relaunch.

## Plugins

Two Tauri plugins, both in `src-tauri/Cargo.toml`:

- `tauri-plugin-updater = "2"` — the check / download / verify / install dance.
- `tauri-plugin-process = "2"` — the `relaunch()` call that restarts the app after install.

Capabilities in `src-tauri/capabilities/default.json`:

```
"updater:default",
"process:default",
"dialog:allow-message"
```

Missing any of these makes the corresponding frontend `invoke` fail with a permission error.

## Endpoint + signing

`tauri.conf.json → plugins.updater`:

```json
{
  "pubkey": "<base64-encoded minisign ed25519 public key>",
  "endpoints": [
    "https://github.com/mosaicvideo/mosaic/releases/latest/download/latest.json"
  ]
}
```

The endpoint is the stable `releases/latest/download/<file>` GitHub alias — it resolves to whichever release is currently marked as "Latest". `latest.json` is generated by `tauri-action` during the release build (see `release.yml`) and attached to every release as an artifact. It contains per-platform asset URLs + minisign signatures:

```json
{
  "version": "0.1.3",
  "notes": "…",
  "pub_date": "2026-04-17T…",
  "platforms": {
    "darwin-universal": { "signature": "…", "url": "…/Mosaic_universal.app.tar.gz" },
    "windows-x86_64":   { "signature": "…", "url": "…/Mosaic_x64-setup.nsis.zip" },
    "windows-aarch64":  { "signature": "…", "url": "…/Mosaic_arm64-setup.nsis.zip" },
    "linux-x86_64":     { "signature": "…", "url": "…/Mosaic_amd64.AppImage.tar.gz" }
  }
}
```

## Key custody

- Private key: `~/.tauri/mosaic-updater.key` locally. In CI, stored as the `TAURI_SIGNING_PRIVATE_KEY` Actions secret.
- Public key: embedded in `tauri.conf.json` (compiled into the binary). Losing the private key orphans every installed copy — they will reject all future updates as unsigned. **Back it up out-of-band.** Rotation requires a synchronous release + manual user download.

## Frontend flow

`src/main.js`:

```js
import { check } from '@tauri-apps/plugin-updater';
import { relaunch } from '@tauri-apps/plugin-process';
import { ask } from '@tauri-apps/plugin-dialog';

async function checkForUpdate() {
  try {
    const update = await check();
    if (!update) return;
    const ok = await ask(
      'Download and install? The app will restart.',
      { title: `Mosaic ${update.version} is available`, kind: 'info',
        okLabel: 'Install', cancelLabel: 'Later' }
    );
    if (!ok) return;
    await update.downloadAndInstall();
    await relaunch();
  } catch (e) {
    // Silent: fails harmlessly in dev mode (no signature), offline, or on rate limits.
    console.warn('update check failed:', e);
  }
}
```

Called fire-and-forget from `init()` alongside `checkTools()` and `createMediaInfoModal()`. Errors are swallowed so:

- Dev-mode launches (no signed bundle) don't spam errors.
- Offline starts are silent.
- GitHub API rate-limiting (60/hr unauth) doesn't break startup.

The dialog title shows the **new** version, not the current one, so the user immediately knows what they're accepting.

## Platform notes

- **macOS:** updates ship as `.app.tar.gz` (not `.dmg`) so tauri can drop-in replace the bundle without user intervention. The app must be in `/Applications` or a writable location; running from the DMG itself fails.
- **Windows:** updates ship as NSIS `.exe.zip`. On install, the installer relaunches the app.
- **Linux:** updates ship as AppImage `.tar.gz`. Same drop-in as macOS.

Each platform's bundle is signed independently. `latest.json` carries the correct signature per variant.

## Release flow interaction

`tauri-action` in `release.yml` is invoked with `createUpdaterArtifacts: true` in `tauri.conf.json → bundle`, which:

1. Builds the normal installer (.dmg / .nsis.exe / .AppImage / .deb / .rpm).
2. Additionally produces the `.app.tar.gz` / `.nsis.zip` / `.AppImage.tar.gz` updater bundle.
3. Signs each with the injected `TAURI_SIGNING_PRIVATE_KEY`.
4. Generates `latest.json` referencing all platform bundles.
5. Uploads everything to the draft release.

**The release must be published**, not left in draft — `releases/latest/download/latest.json` only resolves for published releases. Unpublished = auto-updater returns "no update" even if a newer tag exists.

## Upgrade path

- v0.1.1 and earlier predate the updater plugin entirely — they have no knowledge of newer versions and must be updated manually once. From v0.1.2 onward, subsequent versions auto-install.
- Downgrading via the updater is not supported (tauri-plugin-updater compares versions and only offers strictly-newer).

## Related

- The update-prompt dialog UX was iterated post-ship: v0.1.2 included the full release-notes body, which produced a wall of markdown in the native dialog. Commit `9bba730` (2026-04-18) dropped the body, leaving just the version title + install prompt. Fix is in `src/main.js` `checkForUpdate`.

## Scope exclusions

- No background / periodic checking — launch-only. Users get the prompt when they open the app, not while they're working.
- No delta / patch updates — full bundle each time. Reasonable at current binary sizes (~15 MB).
- No staged rollouts — every user sees the latest release immediately after publish.
- No release-channel support — single stable channel.
